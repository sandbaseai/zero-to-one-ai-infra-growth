# 从 0 到 1 搭建 AI Infra 增长系统

> SandBase.ai 30 天公开运营实战：从一个网站，搭到一个可信的 AI 基础设施品牌。

[English](README.md)

SandBase.ai 是面向生产级 AI agents 的 agent infrastructure 平台。

这个仓库记录的不是产品代码，而是产品背后的运营基础建设：SEO、Blog、社媒账号、开发者社区、GitHub 信任资产、内容分发，以及一个早期技术产品如何从外部看起来真实、清楚、可信。

## 这是什么

这不是普通 SEO 清单，也不是目录提交流水账，更不是一堆 AI 生成的营销稿。

它记录的是一个早期 B2B AI infrastructure 产品如何从 0 开始建立第一层可信度：

- 官网 SEO 和可抓取性
- Google Search Console 验证
- 技术博客和内容基础设施
- X / Discord / LinkedIn / GitHub 基础账号
- 开发者社区入口
- 高质量外链资产
- 每日运营和问题记录
- 使用 Codex 作为 AI 运营负责人/协作伙伴的真实过程

目标很简单：

```text
在向市场要注意力之前，先让 SandBase 看起来真实、有用、有技术可信度。
```

## 北极星

```text
The infrastructure layer for developers building production AI agents.
```

所有内容、账号、外链、社交、社区和官网页面，都围绕这个定位展开。

如果一个动作不能帮助开发者、Google 或潜在客户理解 SandBase 是一个严肃的 agent infrastructure 平台，就不优先做。

## 增长底层逻辑

这个项目的核心逻辑很简单：

```text
产品第一。
先打基础，再做放大。
每天做一点，等待量变产生质变。
增长方案必须从产品自身长出来，而不是套模板。
```

SandBase 不是泛 AI 工具站，而是开发者基础设施产品。所以它的增长方式也不能照搬普通 AI 工具的玩法。

我们更看重：

- 清楚的技术定位
- 可抓取、可信、稳定的公开页面
- 对开发者有帮助的 Docs、Blog 和 GitHub 资产
- 能承接真实问题的社区入口
- 指向产品能力和技术价值的外部分发

目标不是制造一波短期热闹。

目标是让产品更容易被理解、更容易被信任、更容易被发现，也更容易被再次想起。每天把这些信号补强一点，长期才会产生复利。

## 为什么要记录

早期技术产品经常遇到一个问题：

产品可能是真的，但外界看不出来。

一个开发者、客户、投资人或 Google crawler 看到公司时，常见缺口包括：

- 定位不清晰
- 网站内容薄
- 没有公司主页
- 没有 GitHub 信任资产
- 没有技术内容
- 没有可信外链
- 没有社区入口
- 看不出团队在持续建设

这个仓库记录我们如何一步步补齐这些信号。

## 怎么读

如果你是创始人，可以把它当成一份可复制的实战 playbook。

如果你是开发者，可以把它当成一个 AI infra 产品建立公开可信度的幕后记录。

如果你正在用 Codex 做运营，可以重点看每次操作里的确认边界：Codex 可以检查、起草、执行，但发布、提交、改权限这类公开副作用仍然需要人确认。


## 每周目标

### 第 1 周：账号和基础环境准备

目标：

把 SandBase 的基础运营环境搭起来，让它看起来真实、可抓取、可访问，并且能开始日常运营。

交付物：

- 官网 SEO 基线检查
- sitemap 和 Search Console 验证
- Blog 技术内容系统
- GitHub organization 和第一个开发者信任资产
- X 账号创建、定位和安全设置
- Discord 社区入口和权限结构
- LinkedIn 公司主页
- 30 天运营文档
- 第一批内容集群规划

成功标准：

- 开发者可以从官网、Blog、GitHub、LinkedIn、X 或 Discord 理解 SandBase 是什么
- 所有公开渠道使用一致定位
- 不暴露私人账号、邮箱、token 或内部管理信息
- 基础盘足够支撑第 2 周的目录提交和日常运营

### 第 2 周：目录提交和日常运营

目标：

开始可控地对外分发，同时形成每日运营、互动和问题记录习惯。

交付物：

- 提交可信开发者目录
- 准备或发布站外技术文章
- 每日 X / LinkedIn / Discord 更新
- 参与相关开发者讨论，不做垃圾链接投放
- 记录用户问题、阻塞点、误解和内容选题
- 持续完善 GitHub resource repo
- 准备 Product Hunt / Hacker News，但不仓促 launch

成功标准：

- 启动 5-10 个可信外部提及、链接或 profile
- 不买垃圾外链，不群发低质量目录
- 至少一个资产对开发者有真实价值
- 每天至少沉淀一个问题、学习或内容想法

### 第 3 周：问题驱动的内容集群

目标：

把第 2 周运营中遇到的问题、异议和重复困惑，转化为结构化内容集群。

交付物：

- Agent runtime 内容
- Sandboxed execution 内容
- Tools / MCP 内容
- Model routing 内容
- Observability、guardrails、production reliability 内容
- Blog、Docs、首页、Discord、落地页之间的内部链接
- 把 Discord / 社交里的问题转成 FAQ、Docs 或 Blog 选题

成功标准：

- 每个核心主题都有多篇支撑内容
- 每篇文章有明确搜索意图和内部链接路径
- Search Console 开始出现相关关键词 impressions
- Blog 和 Docs 像是在回答真实开发者问题，而不是随机 AI 新闻

### 第 4 周：权威内容和转化

目标：

把前三周的基础、分发和问题发现，转化为权威页面、强 CTA 和外联机会。

交付物：

- `/agent-infrastructure` 落地页
- `/ai-agent-sandbox` 落地页
- Agent Infrastructure Stack pillar 长文
- 面向 newsletter、技术博主、开源项目和生态作者的外联
- 优化 Discord、Docs、Contact、Quickstart 等转化入口

成功标准：

- 至少 3-5 个清晰转化入口
- Pillar 内容成为内部链接 hub
- 外联具体、有上下文，不是群发模板
- 官网、Blog、GitHub、社交、社区的定位一致

## 原则

1. 先建设真实资产，再去要注意力。
2. 技术深度优先于泛 AI 内容。
3. 外链是信任资产，不是数字游戏。
4. 不买垃圾外链。
5. 不群发低质量目录。
6. 不把每篇文章都写成广告。
7. 先参与社区，再分享链接。
8. 记录决策，不只记录任务。
9. 保护私人账号和敏感信息。
10. 使用 AI 协作时保留确认边界。

## Codex 的角色

Codex 在这个项目里不是只写代码，而是 AI 运营协作伙伴：

- 审查网站结构
- 起草和编辑技术内容
- 辅助浏览器操作
- 创建 GitHub 资产
- 记录每日运营日志
- 把零散操作沉淀成 playbook
- 检查公开内容是否泄露私人账号信息

重要边界：

Codex 不会在没有确认的情况下提交表单、发布帖子、添加管理员或修改公开权限。

工作流是：

```text
人设定目标
  ↓
Codex 检查上下文
  ↓
Codex 起草 / 准备 / 操作
  ↓
人确认公开副作用
  ↓
Codex 记录结果
```

这个确认边界，是本项目非常重要的一条经验。
