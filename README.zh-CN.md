# Global AI Cold Start: 30 Days to Win Overseas AI Builders

> 一个实时 build-in-public 增长 playbook：从 0 开始拿下第一批海外 AI builders、可信渠道、内容资产、外链和社区信号。

[English](README.md)

很多 AI startup 不是输在做不出 demo。

而是输在：团队之外的人发现不了、看不懂、不信任，也记不住。

这个仓库记录 SandBase.ai 的出海冷启动增长过程。SandBase.ai 是面向生产级 AI agents 的 agent infrastructure 平台。

Cold Start 的意思是：几乎没有受众、没有搜索基础、没有可信外链、没有开发者社区时，如何一步步让海外 AI builders 发现你、理解你、信任你。目标不是一夜爆红，而是把公开可信度一点点搭起来。

30 天里，我们把一个真实 AI infrastructure startup 变成公开增长案例。

我们正在搭建：

- 让 Google 理解产品
- 让开发者信任产品
- 在 launch 前搭好公开渠道
- 发布能长期复利的技术内容
- 不靠 spam 获取外部提及
- 把每日运营沉淀成可复用 playbook

你可以直接复用：

- 每日增长 checklist
- 增长、海外拓展和黑客松赞助的每日执行面板
- SEO 和内容结构
- 开发者社区搭建方式
- GitHub 信任资产案例
- 目录和外链筛选决策
- Dev.to、X、LinkedIn、Discord 分发记录
- 海外 builder 拓展和黑客松赞助 playbook

## 这是什么

这不是普通 SEO 清单，也不是目录提交流水账，更不是一堆 AI 生成的营销稿。

它是一份面向 AI 创业出海的真实增长日记。

案例是 SandBase.ai，一个 B2B AI infrastructure 产品。可复用的部分适合任何早期 AI 创业者、独立开发者或技术出海团队：

- 官网 SEO 和可抓取性
- Google Search Console 验证
- 技术博客和内容基础设施
- X / Discord / LinkedIn / GitHub 基础账号
- 开发者社区入口
- 高质量外链资产
- 每日运营和问题记录
- 使用 Codex 作为 AI 运营负责人/协作伙伴的真实过程

目标很简单：

```text
在向市场要注意力之前，先让一个 AI startup 看起来真实、有用、有技术可信度。
```

## 北极星

```text
The infrastructure layer for developers building production AI agents.
```

所有内容、账号、外链、社交、社区和官网页面，都围绕这个定位展开。

如果一个动作不能帮助开发者、Google 或潜在客户理解 SandBase 是一个严肃的 agent infrastructure 平台，就不优先做。

## 增长底层逻辑

这个项目的核心逻辑很简单：

```text
产品第一。
先打基础，再做放大。
每天做一点，等待量变产生质变。
增长方案必须从产品自身长出来，而不是套模板。
```

SandBase 不是泛 AI 工具站，而是开发者基础设施产品。所以它的增长方式也不能照搬普通 AI 工具的玩法。

我们更看重：

- 清楚的技术定位
- 可抓取、可信、稳定的公开页面
- 对开发者有帮助的 Docs、Blog 和 GitHub 资产
- 能承接真实问题的社区入口
- 指向产品能力和技术价值的外部分发

目标不是制造一波短期热闹。

目标是让产品更容易被理解、更容易被信任、更容易被发现，也更容易被再次想起。每天把这些信号补强一点，长期才会产生复利。

## 为什么要记录

早期技术产品经常遇到一个问题：

产品可能是真的，但外界看不出来。

一个开发者、客户、投资人或 Google crawler 看到公司时，常见缺口包括：

- 定位不清晰
- 网站内容薄
- 没有公司主页
- 没有 GitHub 信任资产
- 没有技术内容
- 没有可信外链
- 没有社区入口
- 看不出团队在持续建设

这个仓库记录我们如何一步步为 SandBase 补齐这些信号，同时把过程沉淀成其他创业者也能复用的增长 playbook。

## 每周目标

### 第 1 周：账号和基础环境准备

目标：

把 SandBase 的基础运营环境搭起来，让它看起来真实、可抓取、可访问，并且能开始日常运营。

交付物：

- 官网 SEO 基线检查
- sitemap 和 Search Console 验证
- Blog 技术内容系统
- GitHub organization 和第一个开发者信任资产
- X 账号创建、定位和安全设置
- Discord 社区入口和权限结构
- LinkedIn 公司主页
- 30 天运营文档
- 第一批内容集群规划

成功标准：

- 开发者可以从官网、Blog、GitHub、LinkedIn、X 或 Discord 理解 SandBase 是什么
- 所有公开渠道使用一致定位
- 不暴露私人账号、邮箱、token 或内部管理信息
- 基础盘足够支撑第 2 周的目录提交和日常运营

### 第 2 周：分发和渠道测试

目标：

开始可控地对外分发，同时形成每日增长动作、真实互动和问题记录习惯。

交付物：

- 提交可信开发者目录
- 准备或发布站外技术文章
- 每日 X / LinkedIn / Discord 更新
- 参与相关开发者讨论，不做垃圾链接投放
- 记录用户问题、阻塞点、误解和内容选题
- 持续完善 GitHub resource repo
- 准备 Product Hunt / Hacker News，但不仓促 launch

成功标准：

- 启动 5-10 个可信外部提及、链接或 profile
- 不买垃圾外链，不群发低质量目录
- 至少一个资产对开发者有真实价值
- 每天至少沉淀一个问题、学习或内容想法

### 第 3 周：开源增长底座

目标：

把 SandBase 的开源资产变成主要信任入口和分发引擎。

交付物：

- 让 `awesome-agent-runtime` 具备外部贡献入口
- 发布或准备 500 项目 agent infrastructure map
- 准备 execution sandbox 内容集群
- 准备 MCP / tool protocol 内容集群
- 准备 maintainer outreach 草稿和互动规则
- 生成可用于社媒分发的 PNG 图片
- 所有公开发布继续保留人工确认边界

成功标准：

- 开源入口项目可以承接社区提交
- Day 18-20 素材都有清晰的发布 checklist
- maintainer 互动具体、有上下文、不像 spam
- repo 和社媒资产能解释 SandBase 的 agent infrastructure 定位，但不过度承诺

### 第 4 周：Launch readiness 和 30 天复盘

目标：

收口第一个冷启动周期，让所有公开资产准备好进入下一阶段。

交付物：

- 清理外部 profile 和 backlink surfaces
- 检查 SandBase.ai 官网、footer、docs、blog、status page、社媒链接
- 准备 Product Hunt 或其他 launch surface 素材，但不仓促 launch
- 启动开源项目社区提交循环
- 收集 metrics、经验、问题和渠道决策
- 起草或发布最终 30 天案例复盘
- 定义下一个 30 天主题：从 public trust 到 developer adoption

成功标准：

- 官网、GitHub、X、LinkedIn、DEV.to、Discord 和外部 profiles 讲同一个故事
- launch 素材真实、克制，可支持 go/no-go 决策
- metrics 有记录，但不夸大
- 最终复盘对其他 founder 或 builder 也有复用价值
- 下一阶段 30 天运营主题清晰

## 原则

1. 先建设真实资产，再去要注意力。
2. 技术深度优先于泛 AI 内容。
3. 外链是信任资产，不是数字游戏。
4. 不买垃圾外链。
5. 不群发低质量目录。
6. 不把每篇文章都写成广告。
7. 先参与社区，再分享链接。
8. 记录决策，不只记录任务。
9. 保护私人账号和敏感信息。
10. 使用 AI 协作时保留确认边界。

## Codex 的角色

Codex 在这个项目里不是只写代码，而是 AI 运营协作伙伴：

- 审查网站结构
- 起草和编辑技术内容
- 辅助浏览器操作
- 创建 GitHub 资产
- 记录每日运营日志
- 把零散操作沉淀成 playbook
- 检查公开内容是否泄露私人账号信息

重要边界：

Codex 不会在没有确认的情况下提交表单、发布帖子、添加管理员或修改公开权限。

工作流是：

```text
人设定目标
  ↓
Codex 检查上下文
  ↓
Codex 起草 / 准备 / 操作
  ↓
人确认公开副作用
  ↓
Codex 记录结果
```

这个确认边界，是本项目非常重要的一条经验。
