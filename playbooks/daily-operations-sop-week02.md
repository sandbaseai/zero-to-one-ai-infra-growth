# Daily Operations SOP — Week 2

Stage: controlled distribution and community learning

Use this SOP once per day during Week 2. The goal is not to create noise. The goal is to make SandBase visible in the places where developers already talk about agents, runtime, tools, sandboxing, distributed compute, and production reliability.

## Daily Outcome

By the end of each day, we should have:

- one product-facing public update or post
- two to five useful interactions across X, Discord, Peerlist, LinkedIn, or developer communities
- one external profile, directory, content asset, or ecosystem asset moved forward
- one question, objection, or insight recorded
- the daily build-in-public notes updated

## Daily Checklist

### 0. Channel Sweep

Time box: 10-15 minutes

Check the channels SandBase already operates:

- X
  - notifications
  - mentions
  - replies to recent posts
  - 2-3 high-quality topic searches
- LinkedIn
  - company page notifications
  - comments on recent posts
  - possible company-page update
- Discord
  - `#agent-runtime`
  - `#builder-chat`
  - any new replies or intros
- Peerlist
  - notifications
  - comments on SandBase profile/posts
  - relevant Scroll discussions
- DevHunt
  - account/profile status
  - tool draft status
  - free listing path or pricing changes

Do not force activity. If a channel has no useful interaction, record "checked, no action".

### 1. Check Search and Site Health

Time box: 10 minutes

- Check Google Search Console for:
  - sitemap status
  - newly indexed pages
  - crawl errors
  - unexpected search appearance issues
- Check whether recently shipped pages appear in sitemaps:
  - Agent Ecosystem
  - Hackathon Support
  - blog articles
  - docs pages
- If something is broken, record it as a P0/P1/P2 issue instead of letting it interrupt the whole day.

### 2. Publish One Product-Facing Thought

Time box: 15 minutes

Pick one channel:

- X for short technical opinions
- LinkedIn for company trust and B2B credibility
- Peerlist for builder-facing notes
- Discord for internal community discussion

Rules:

- no hard selling
- no repeated links
- no generic AI hype
- make one clear technical point
- connect the point to production agents, runtime, tools, sandboxing, model routing, distributed compute, or reliability

Good format:

```text
Observation

Why it matters

Concrete infrastructure angle
```

### 3. Do Real Community Interaction

Time box: 20-30 minutes

Target count:

- minimum: 2 useful interactions
- normal: 3-5 useful interactions
- maximum: stop before it feels forced

Good interaction targets:

- builders showing agent projects
- discussions about agent runtime
- tool calling failures
- sandboxing, browser, shell, file access
- distributed compute for agents
- observability, evaluation, reliability
- emerging agent infrastructure projects

Avoid:

- AI income threads
- prompt-course threads
- roadmap bait
- generic "what are you building" threads with no substance
- copy-paste replies

Interaction standard:

```text
specific context -> engineering observation -> useful next question or proof point
```

### 4. Move One Distribution Asset Forward

Time box: 20-40 minutes

Choose one:

- finish one directory/profile draft
- prepare one launch asset
- improve one GitHub README
- add one useful project to Emerging Agent Infrastructure registry
- prepare one off-site article outline
- improve one docs / blog / landing page from real feedback

Do not submit paid launches above the budget cap without a separate decision.

### 5. Record Learnings

Time box: 10 minutes

Update the daily note with:

- what was posted
- where we interacted
- links
- what people asked or ignored
- blockers
- one lesson
- tomorrow's next action

The daily note should be useful even if someone reads it one month later.

```text
Make each one useful.

```
