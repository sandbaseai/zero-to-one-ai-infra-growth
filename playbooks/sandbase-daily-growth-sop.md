# SandBase Daily Growth SOP

Date: 2026-06-25

This is the primary daily operating SOP for SandBase.ai during the 30-day growth plan.

Use this before opening any channel. The goal is to keep daily work focused, useful, and repeatable instead of drifting between platforms.

## Quick Start Card

Copy this into the current day log before starting:

```md
Today's objective:

Main asset:

Channels to check first:
- GitHub:
- X:
- LinkedIn:
- DEV.to:
- Discord:

Public actions that need confirmation:

Do not do today:

End state:
- [ ] one main asset prepared or published
- [ ] useful interactions completed
- [ ] links and metrics recorded
- [ ] tomorrow's next action written
```

## Operating Principle

```text
One day, one main asset, several useful interactions, one clean log.
```

Every public action should strengthen at least one of these signals:

- SandBase.ai is a serious agent infrastructure company.
- SandBase understands production agent runtime problems.
- SandBase open-source assets are useful to builders.
- Developers can discover, trust, and revisit SandBase.

## Daily Order

Do the steps in this order.

## 1. Pick Today's Main Objective

Open:

- [30-Day Completion Plan](../30-day-growth-diary/30-day-completion-plan.md)
- the current day file under `30-day-growth-diary/days/`

Write down one sentence:

```text
Today's objective is:
```

Examples:

- announce the 500-project agent infrastructure map
- publish a sandbox-runtime checklist
- clean up external profiles
- prepare Product Hunt launch assets
- collect metrics and write the recap

If the day has too many tasks, choose one primary output and make everything else support it.

## 2. Check Existing Signals First

Before posting anything new, check whether people already replied.

Priority order:

1. GitHub issues, PRs, discussions, and comments already touched by SandBase.
2. X notifications and relevant replies.
3. LinkedIn company page comments and post reactions.
4. DEV.to article comments.
5. Discord public channels.
6. Peerlist or other profile surfaces, only if the browser session is stable.

Record:

- real replies
- maintainer responses
- useful questions
- objections
- broken links or profile problems

Skip:

- vanity-only checks with no action
- repeated refresh loops
- blocked pages that do not become usable after one reasonable attempt

## 3. Publish Or Prepare One Main Asset

Each day should produce one main asset.

Asset types:

- X post or thread
- LinkedIn company-page post
- DEV.to / Hashnode article
- GitHub issue/discussion
- README or docs improvement
- generated PNG image
- launch/profile copy
- final recap section

Public side-effect rule:

- Drafts, images, docs, and local repo changes can be prepared directly.
- Public posts, public comments, paid submissions, and launch actions need human confirmation.

For each main asset, record:

- title or short description
- file path or public URL
- channel
- image used, if any
- whether it is drafted, published, or waiting for confirmation

## 4. Do Targeted Interactions

Interactions should be useful even if nobody clicks a link.

Daily targets:

| Channel | Normal Target | Hard Limit |
| --- | ---: | ---: |
| GitHub | 1-3 comments or discussions | 5 |
| X | 5-15 replies/follows | 25 |
| LinkedIn | 3-10 comments/replies | 15 |
| Discord | 0-2 useful messages | 3 |
| DEV.to | reply only when there are comments | 3 |

Quality gate before every interaction:

- Is this about agent infrastructure, sandboxing, runtime, tools, memory, evals, observability, deployment, or developer trust?
- Can the message stand alone without a SandBase link?
- Is it specific to the project, post, issue, or person?
- Would the receiver feel respected rather than targeted?

If any answer is no, skip.

## 5. Link Policy

Use links carefully.

Prefer linking to:

1. `awesome-agent-runtime` when the topic is ecosystem mapping.
2. a SandBase blog or DEV.to article when the topic matches the article.
3. SandBase.ai when the conversation is about the product, company, or infrastructure platform.

Do not link when:

- the reply is only a lightweight opinion
- the thread did not ask for resources
- the same link was already posted in that community recently
- the comment would look like a backlink attempt

Maintainer inclusion note:

```text
We included this project in Awesome Agent Runtime under <category>:
https://github.com/sandbaseai/awesome-agent-runtime

Happy to adjust the description if there is a better positioning.
```

Use the note only when:

- the project is actually included
- the category is correct
- the context is relevant
- the comment is not being posted only for a backlink

## Channel Playbooks

## GitHub

Best use:

- open-source trust
- maintainer relationships
- ecosystem corrections
- technical discussions
- backlinks that are earned through useful participation

Daily GitHub loop:

1. Check replies on previous comments.
2. Check `awesome-agent-runtime` issues and PRs.
3. Pick 1-3 high-fit projects from the outreach shortlist.
4. Read the issue/discussion before commenting.
5. Comment only with a specific technical angle.
6. Record the URL and follow-up date.

Good GitHub comment patterns:

- offer a validation angle
- suggest a small checklist
- ask for preferred project positioning
- provide a correction with source
- connect an issue to a production-agent runtime concern

Bad GitHub comment patterns:

- "Cool project, check us out"
- generic inclusion messages on inactive issues
- asking for stars
- posting the same wording across repos

## X

Best use:

- daily developer visibility
- short technical opinions
- following relevant builders
- replying to agent/runtime/tooling conversations

Daily X loop:

1. Check notifications.
2. Search one active keyword.
3. Reply to 5-10 relevant posts.
4. Follow 5-20 relevant builders or maintainers.
5. Publish one main post only if there is a real asset or idea.

Preferred post types:

- short technical framing
- open-source milestone
- category map
- checklist
- "what we learned" thread

Avoid:

- generic AI hype
- repeated link posts
- high-volume following of broad AI accounts

## LinkedIn

Best use:

- company credibility
- B2B infrastructure narrative
- founder/operator trust
- visual summaries

Daily LinkedIn loop:

1. Check company-page comments.
2. Reply to any real comments.
3. Publish at most one company-page post.
4. Comment thoughtfully on 3-10 relevant posts.
5. Use images for milestone or architecture posts.

LinkedIn tone:

- clear
- practical
- slightly broader than GitHub
- no meme-heavy language
- no exaggerated claims

## DEV.to / Long-Form

Best use:

- durable external article links
- technical explanation
- category education

Rules:

- publish only when the post teaches something useful
- use canonical links when republishing SandBase blog content
- mention SandBase at the end unless product context is necessary earlier
- use generated PNGs only when they clarify the article

## Discord

Best use:

- early feedback
- builder questions
- lightweight community rhythm

Rules:

- question-first, link-second
- do not post if there is no real angle
- avoid repeating social posts as Discord messages

## Directory / Launch Sites

Best use:

- selected backlinks
- product discovery surfaces
- launch preparation

Rules:

- no paid submissions without approval
- no broad directory spam
- record blocked pages instead of fighting them
- only submit when the profile copy is accurate and the surface is credible

## Stop Conditions

Stop the daily session when any of these is true:

- one main asset is prepared or published
- at least five useful interactions are completed
- one external link or public signal is recorded
- the current day log is updated
- the next action is clear

Do not keep browsing just to feel busy.

## Weekly Review

Every 7 days, summarize:

- assets published
- public links created
- GitHub stars and issues
- X follower and reply movement
- LinkedIn follower and impression movement
- best questions from builders
- weakest channel
- next week's primary category

Decision rule:

```text
Double down on channels that produce replies, corrections, or useful questions.
Pause channels that only produce vanity impressions.
```
